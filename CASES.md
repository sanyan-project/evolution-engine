# 工控排障案例 — 进化引擎应用示范

> 以下案例展示进化引擎如何应用于真实的工控排障场景。
> 来自黄sir（3年德克威尔FAE）的现场实战经验。

---

## 案例1：远程IO模块ERR灯闪烁

### 现场现象
- 远程IO模块电源灯（PWR）正常
- ERR灯不规则闪烁
- 数据时断时续
- 换过模块、查过接线，问题依旧

### 传统排查（人）
1. 万用表量端子电压 → 正常
2. 换一根网线 → 没好
3. 进TIA Portal看诊断信息 → 模棱两可
4. 怀疑EMC干扰 → 查了很久

排查时间：**2-3小时**，结果是——24V电源线拉了40米到末端只剩16V。

### 引擎怎么帮忙
1. 输入：现象描述("ERR闪烁+PWR正常")
2. 引擎的6D基因组在"排障维度"上搜索：
   - 供电维：电压正常否？
   - 通信维：LINK灯状态？
   - 组态维：型号匹配否？
   - IP维：地址冲突否？
3. 输出：按fitness排序的排查建议列表

### 引擎的优势
- 不会只盯着网线，同时查5个维度
- 从过往案例中自动学习：供电问题是ERR灯闪烁最可能的根因（排名第一）
- 11000+代进化积累的"经验"辅助决策

## 案例2：Profinet通信时好时坏

### 现场现象
- PLC和远程IO之间通信不稳定
- 有时正常跑一整天，有时十分钟断三次
- 断的时候没有任何报警
- LINK灯一直常亮（看起来正常）

### 传统排查（人）
1. 换网线 → 没好
2. 换交换机端口 → 好了两天，又犯
3. 怀疑交换机 → 换了个新的 → 还是偶尔犯
4. 终于发现：网线和变频器输出线在同一个线槽里走了十几米

排查时间：**断断续续一周**。

### 引擎怎么帮忙
1. 输入："通信间歇中断 + LINK灯正常 + 无报警"
2. 引擎在KG中匹配类似模式：
   - 模式A：LINK正常但不稳定 → 排除物理层故障
   - 模式B：间歇性+无关联报警 → 方向→环境干扰
   - 模式C：变频器附近的通信故障 → 已记录3个类似案例
3. 输出：优先查布线环境（动力线/信号线间距）

### 引擎的优势
- 知识图谱中已有"变频器干扰通信"的记录
- 能从"LINK灯正常但通信断"这个矛盾信息中推断出干扰因素
- 比人更快地从海量历史案例中找到相似模式

## 案例3：远程IO模块选型

### 现场需求
- 某自动化产线需要120路DI + 80路DO
- 现场空间有限
- 预算有上限
- 需要PROFINET通讯

### 传统方法（人）
1. 翻开德克威尔产品手册
2. EC-T2008 → 8通道，不够
3. EC-T2016 → 16通道，需要8个才能满足DI，占空间
4. EC-D1616 → 16路DI/DO混合模块，更灵活
5. 反复计算通道数、价格、安装空间、总线负载

选型时间：**半天到一天**。

### 引擎怎么帮忙
1. 输入约束：120DI/80DO/空间/预算/PROFINET
2. 引擎的6D基因组在"选型维度"上搜索：
   - 功能维：满足通道数
   - 经济维：在预算内
   - 空间维：安装占位最小
   - 可靠性维：MTBF历史数据
3. 输出：Pareto最优的3-5种组合方案

### 引擎的优势
- 同时优化5个维度的参数，人一般只能考虑2-3个
- 不受思维定式限制——人倾向于推荐自己熟悉的型号
- 从历史项目中学习哪种组合实际运行最稳定

---

## 总结

这三个案例不是"引擎已经完美解决"的意思，而是展示：

1. **工控排障的本质是一个多维搜索问题**——正好是进化引擎擅长的
2. **人的经验是有限的**（你能记住的案例就那么多），但引擎的KG可以无限积累
3. **引擎的价值不是代替FAE**，而是辅助FAE更快、更全面地做决策
